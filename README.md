## 概要
Web系エンジニアです。現在は、フリーランスとして活動。

AWSを利用したインフラ構築、監視基盤の構築、バックエンドをメインとしています。

## 強み
- AWSを利用したインフラの設計・構築・運用
- Datadog等を利用した監視体制の設計・開発・運用
- Terraformを利用したIaCによるインフラ開発の効率化、自動化

Web開発においてインフラの整備からバックエンドアプリケーションの開発までの工程を担当できます。

新規開発、サービスのリプレイスフェーズにおいて最もバリューを発揮出来るかと思います。

3名

- インフラの改善・運用を担当

#### 状況・課題/工夫した取り組み・自身が意思決定したこと/成果

- 状況
  - インフラのメンテナンスに手が回っておらず、長期間放置された状態だった。不要なIAMやEC2が残っていたり、過剰なサイズのインスタンスが稼働していたりと、リソースの管理が行き届いていなかった。また、TerraformのStateが肥大化しており実行に時間がかかる状態だった。Terraform workspaceが使われておりコードの可読性が低く、事故が起きやすい状況だった。
- 行動・判断
  - まず、全体的に何が利用されているかを確認し、不要なリソースの削除から着手した。過剰なサイズのEC2インスタンスは適切なサイズに変更し、コストを削減した。
  - 肥大化したTerraformのStateを適切な単位に分割し、開発効率を改善した。
  - Terraform workspaceをやめ、環境ごとにディレクトリを分離することで、安全にTerraformを運用できるように修正した。
  - Terraform fmtを実行するCIを導入し、コードの品質を維持できるようにした。
  - 長期間更新されていなかったTerraformおよびAWS Providerのバージョンアップを一人で実施した。
- 成果
  - 不要リソースの削除とインスタンスサイズの適正化により、AWSコストを削減した。
  - TerraformのState分割と環境ごとのディレクトリ分離により、開発効率と運用の安全性が向上した。
  - CIの導入とバージョンアップにより、コード品質の維持と最新機能の利用が可能になった。

### 金融系企業でAWSを中心としたインフラの構築、監視基盤の構築 (2019-2022)

#### 業務概要
AWSインフラの設計・構築・運用、およびDatadogを利用した監視基盤の構築

#### 利用した技術
- AWS（ECS Fargate、StepFunctions、Lambda、ALB等）
- Datadog
- Terraform
- Go

#### プロジェクトの規模
2名

#### 自分の役割
メンバーとして参画したが、インフラ領域は実質的に一人で担当し、設計から構築・運用まで主導した

#### 状況・課題/工夫した取り組み・自身が意思決定したこと/成果

- 状況
  - 監視基盤が存在せず、サービスの異常を検知できない状態だった
  - EC2上でアプリケーションとバッチが稼働しており、運用負荷が高くAWSコストも増大していた
- 行動・判断
  - **監視基盤の構築**：Datadogによる外形監視と、エラーログをチャットに通知する仕組みを構築し、サービスの異常を即座に検知・共有できる体制を整備した
  - **EC2からECS Fargateへの移行**：サーバー管理が不要になることで運用負荷を軽減できると判断し、Fargateを選定。インフラ設計、監視設定、性能試験、チューニング、CI/CDの構築を担当。無停止によるリリースを行い、ユーザー影響なく移行を完了させた
  - **バッチ処理のStepFunctions + ECS Fargate移行**：StepFunctionsを採用した理由は、ワークフローによりエラー箇所が特定しやすく、リトライ処理が容易な点。Lambdaによるエラーハンドリングを導入し、AWS側の障害時にも自動で再実行する仕組みを構築した。開発初期段階でCI/CDを導入し、チームの開発速度向上に貢献した
  - **運用改善**：セキュリティ強化、IAMアカウント管理の見直し、コスト削減のための構成見直しを実施
  - **ナレッジ共有**：Terraformの社内向け勉強会を開催。録画を残すことで新規メンバーのキャッチアップ環境も整備した
- 成果
  - 監視基盤の構築により、サービスの異常に即座に対応できる体制を確立
  - EC2からFargateへの移行により、EC2の運用コストとAWSコストを削減
  - バッチ処理の移行により、常時起動から必要時のみ起動する構成に変更し、コストを削減。自動再実行の仕組みにより安定したバッチ実行環境を実現
  - ECS移行の知見を先行してチームに共有し、後続の移行作業の開発速度を向上させた
  - 勉強会によりインフラ担当者以外もTerraformを利用できるようになり、属人化を軽減
